# 19 — Internationalization

## Scope

- **Code, identifiers, file names, comments, log messages, error codes:** English only.
- **UI strings, email templates, AI system prompts:** translatable via catalogs.
- **Data:** stored as-is in the tenant's chosen locale; no engine-level
  translation of business records.

## Default and fallback

- Default UI locale: `en`.
- Fallback chain: requested → tenant default → `en`.

## Locale negotiation

Order of precedence:

1. `?locale=` query param (per-request override).
2. `Accept-Language` header.
3. User profile setting (`users.locale`).
4. Tenant default (`ir_config_param: "i18n.default_locale"`).
5. Engine default `en`.

## Frontend strategy

- Catalogs live in `packages/ui/src/locales/<locale>/<domain>.json`.
- Format: ICU MessageFormat for plurals and gender.
- Hook: `useT()` from `@orbiteus/ui` returns a translator scoped to the loaded
  catalogs.
- Lazy-load catalogs on locale change.

## Backend strategy

- Error codes are stable identifiers (`auth.invalid_credentials`).
- `detail` strings can be localized; consumers should rely on `code` for logic.
- Email and notification templates use Jinja2 with locale-keyed includes.

## Custom field translations

- `custom_fields_def` declares per-key labels, types, and translations.
- API returns labels in the negotiated locale.
- Stored as `ir_translation` rows for engine-level strings (menus, form labels,
  AI suggested prompts, action labels).

## Date and number formats

- Use `dayjs` with locale plugin on the frontend.
- Backend returns ISO 8601 timestamps; formatting is a frontend concern.
- Currency formatting is locale + tenant currency; see canonical CRM doc for
  the `monetary` widget.

## What you do not do

- Do not store locale-specific business data in mapping columns
  (`name`, `description`). Use `ir_translation`.
- Do not hardcode UI strings in TSX outside the catalog.
- Do not pass locale-specific text to AI models — pass keys + locale, let
  the prompt include the rendered translation.
